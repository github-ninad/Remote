# Contributing to REMOTE

Contributions are welcome. The model gets better when more people use it, break it, and PR fixes back.

This is a remote-work playbook — the contribution process matches the principles inside it.

## How to Contribute

1. **Fork** the repo
2. **Create a branch** — name it for the change (`add-onboarding-template`, `fix-phase2-pitfall`, etc.)
3. **Make your changes**
4. **Open a PR** — write the description as "say why, not just what." The diff shows *what* changed; the description explains *why* it should. If you're proposing a new ritual or template, explain the failure mode it addresses.
5. **Tag at least two reviewers**
6. **Discussion happens async in PR comments** — default response window is 72 hours. If you need a decision faster, say so and explain why.

## What Makes a Good Contribution

- A real-world correction: something you ran, discovered was wrong or incomplete, and fixed
- A missing ritual or template that addresses a documented failure mode
- A cleaner framing of something that's accurate but confusing
- An anti-pattern you encountered that isn't listed

## What to Avoid

- Watering down opinionated language without a clear reason — the directness is intentional
- Adding tools to the Recommended Tooling Stack without explaining the use case
- Generic suggestions without grounding in a specific phase or failure mode

## Versioning

This playbook follows semver:
- **Minor** (`v2.1`, `v2.2`): new sections, new templates, significant refinements
- **Major** (`v3.0`): fundamental changes to the phase model or core principles
